## 背景

技能已成为 Claude Code 中最常使用的扩展点之一。它们灵活、易于制作且简单分布。但这种灵活性也使很难知道什么最有效。Thariq 分享了 Anthropic 在数百个活跃使用中广泛使用技能所学的课程。

## 什么是技能？

一个常见的误解是技能是"仅 markdown 文件"，但最有趣的部分是它们是**文件夹**可以包括脚本、资源、数据等 — 代理可以发现、探索和操作的东西。技能也有广泛的配置选项，包括注册动态钩子。

---

## 技能的类型

在分类所有他们的技能后，团队注意到它们聚集成 9 个重复类别。最好的技能整洁地适应一个；更混淆的在几个之间跨越。

### 1/ 库和 API 参考

解释如何正确使用库、CLI 或 SDK 的技能。这些可能是内部库或 Claude Code 有时遇到麻烦的常见库。它们经常包括参考代码片段文件夹和避免编写脚本时陷阱列表。

**示例：** billing-lib、internal-platform-cli、frontend-design

### 2/ 产品验证

描述如何测试或验证你的代码工作的技能。这些经常与 Playwright、tmux 等外部工具配对。验证技能对确保 Claude 输出正确非常有用。

**示例：** signup-flow-driver、checkout-verifier、tmux-cli-driver

### 3/ 数据获取和分析

连接到你的数据和监控堆栈的技能。这些可能包括库以获取带凭证的数据、特定仪表板 ID 等，以及关于常见工作流或获取数据方式的说明。

**示例：** funnel-query、cohort-compare、grafana

### 4/ 业务流程和团队自动化

自动重复工作流进入一个命令的技能。这些通常是相当简单的说明但可能有更复杂的其他技能或 MCP 的依赖。保存前一次执行的结果在日志文件中可帮助模型保持一致并反映之前执行的关于工作流的想法。

**示例：** standup-post、create-<ticket-system>-ticket、weekly-recap

### 5/ 代码搭建和模板

为代码库中特定函数生成框架样板的技能。你可能结合这些技能与可组成的脚本。当你的搭建有原生语言无法纯粹覆盖的要求时，它们特别有用。

**示例：** new-<framework>-workflow、new-migration、create-app

### 6/ 代码质量和审查

在你的组织内执行代码质量并帮助审查代码的技能。这些可以包括确定性脚本或工具以获得最大稳健性。你可能想作为 git 钩子或 GitHub 操作的一部分自动运行这些技能。

**示例：** adversarial-review、code-style、testing-practices

### 7/ CI/CD 和部署

帮助你获取、推送和部署代码到代码库内的技能。这些技能可能参考其他技能以收集数据。

**示例：** babysit-pr、deploy-<service>、cherry-pick-prod

### 8/ Runbooks

采用症状（如 Slack 线程、警报或错误签名）、遍历多工具调查并生成结构化报告的技能。

**示例：** <service>-debugging、oncall-runner、log-correlator

### 9/ 基础设施操作

执行常规维护和操作程序的技能 — 其中一些涉及受益于护栏的破坏性操作。这些使工程师更容易在关键操作中遵循最佳实践。

**示例：** <resource>-orphans、dependency-management、cost-investigation

## 制作技能的提示

9 个最佳实践用于编写有效技能，加上分发和测量的指导。

### Tip 1: 不要陈述明显

Claude Code 了解你的代码库很多，Claude 了解很多关于编码的，包括许多默认意见。如果你发布一项主要是关于知识的技能，尽量专注于将 Claude 推出其常规思维方式的信息。前端设计技能是一个很好的例子 — 它通过与客户迭代来构建以改进 Claude 的设计品味，避免像 Inter 字体和紫色渐变这样的经典模式。

### Tip 2: 构建一个陷阱部分

任何技能中最高信号的内容是陷阱部分。这些部分应该从 Claude 在使用你的技能时遇到的常见故障点构建。理想情况下，你会随着时间推移更新你的技能以捕获这些陷阱。

### Tip 3: 使用文件系统和渐进披露

一项技能是一个文件夹，不仅仅是 markdown 文件。你应该想象整个文件系统作为上下文工程和渐进披露的一种形式。告诉 Claude 你的技能中有什么文件，它会在适当的时间读取它们。最简单的形式是指向其他 markdown 文件 — 例如，将详细函数签名和使用示例拆分为 `references/api.md`。你可以有参考、脚本、示例等的文件夹。

### Tip 4: 避免导向 Claude

Claude 通常会尽力坚持你的说明，由于技能是如此可重用，你会想小心不要太具体。给 Claude 需要的信息，但给它灵活适应情况。不是规定性逐步说明，给目标和约束。

### Tip 5: 考虑设置

一些技能可能需要通过用户上下文进行设置。一个好模式是在技能目录中的 `config.json` 文件中存储此设置信息。如果配置未设置，代理可以然后询问用户信息。你可以指示 Claude 使用 AskUserQuestion 工具进行结构化、多选问题。

### Tip 6: 描述字段用于模型

当 Claude Code 启动会话时，它构建每个可用技能及其描述的列表。此列表是 Claude 扫描以决定"是否有此请求的技能？"的。这意味着描述字段不是总结 — 它是描述**何时触发**此技能。为模型编写它。

### Tip 7: 内存和存储数据

一些技能可以通过存储其中的数据包括一种内存形式。你可以将数据存储在简单的东西中，如附加只日志文件或 JSON 文件，或 SQLite 数据库之类的复杂东西。存储在技能目录中的数据在你升级技能时可能被删除，所以使用 `${CLAUDE_PLUGIN_DATA}` 作为每个插件的稳定文件夹在其中存储数据。

### Tip 8: 存储脚本和生成代码

你能给 Claude 的最强大工具之一是代码。给 Claude 脚本和库让 Claude 花费其回合组成，决定下一步做什么，而不是重建样板。Claude 然后可以即时生成脚本以组成此功能以进行更高级分析。

### Tip 9: 按需钩子

技能可以包括仅在技能被调用时激活的钩子，并在会话持续时间内持续。为更强硬的钩子使用此，你不想一直运行但有时非常有用。

**示例：**
- `/careful` — 通过 PreToolUse 匹配器在 Bash 上阻止 rm -rf、DROP TABLE、force-push、kubectl delete
- `/freeze` — 阻止任何不在特定目录中的编辑/写入

## 分发技能

两种方式与你的团队共享技能：
- **检入你的存储库**（在 `.claude/skills` 下） — 最适合在相对较少的存储库上工作的较小团队
- **制作一个插件**并让 Claude Code 插件市场，用户可以上传和安装插件

已检入的每项技能也会添加一点到模型的上下文中。随着你扩展，内部插件市场允许你分发技能并让你的团队决定要安装哪些。

---

## 衡量技能

为了理解技能的情况，使用预工具钩子让你在公司内记录技能使用。这意味着你可以找到受欢迎的技能或与期望相比触发不足的技能。

---

## 结论

技能是不可思议的强大、灵活工具用于代理，但仍然早期，我们都在弄清楚如何最好地使用它们。想象这个更像是有用提示的抓包而不是明确的指南。最好的方式理解技能是开始、实验并看到对你有效的。大多数我们的始于几行和单个陷阱，随着人们因为 Claude 击中新的边界情况而继续添加它们而变得更好。
